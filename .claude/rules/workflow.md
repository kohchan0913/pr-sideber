# 開発ワークフロー

## フロー概要

```
ユーザーが Issue 指示
  ↓
/do で自律ワークフロー開始
  ↓
[Phase 1] 準備 (ブランチ作成、Issue 確認)
  ↓
[Phase 2] 計画 (planner agent)
  ↓
[Phase 3a] テスト作成 (implementer agent — RED フェーズのみ)
  ↓
[Phase 3b] テストレビュー (quality-reviewer — テスト設計の妥当性検証)
  ↓
[Phase 3c] 実装 (implementer agent — GREEN → REFACTOR)
  ↓
[Phase 4] レビュー (4 reviewer agents 並列)
  ↓
[Phase 5] 修正 (implementer agent) ← 最大3回
  ↓
[Phase 6] PR 作成 & 実施報告
```

## メインコンテキストの役割

IMPORTANT: メインコンテキストは実装しない。以下に専念する。

- 全体のとりまとめ (Phase 間の橋渡し)
- 各 agent の成果物のサニティチェック
- ユーザーとのコミュニケーション
- ワークフローの異常検知

## サニティチェック項目

### Phase 2 (計画) 後
- 責務分担 (TS/Rust/Svelte) に違反していないか
- ファイルパスが実在するか
- 計画の粒度が十分か
- テスト計画がカバレッジ目標を満たせるか

### Phase 3a (テスト作成) 後
- テストが網羅的に書かれているか (エッジケース含む)
- テストが全て RED (失敗) 状態であるか
- プロダクションコードが含まれていないか

### Phase 3b (テストレビュー) 後
- レビュー指摘がテスト設計に反映されたか
- テストケースの漏れが補完されたか

### Phase 3c (実装) 後
- ビルドが通るか (`pnpm build` + `cargo clippy`)
- テストが全て GREEN (通過) 状態であるか
- サイレントフォールバックがないか
- 計画からの逸脱がないか

## 異常検知

以下の場合はワークフローを停止してユーザーに報告する:
- planner が「この Issue は曖昧で計画を立てられない」と判断した場合
- implementer が「計画にない設計変更が必要」と報告した場合
- レビューで CRITICAL が出た場合
- 修正ループが3回に達した場合
- テストが通らない状態が続く場合
